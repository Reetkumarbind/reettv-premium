# Video Playback Flow Analysis - Channel Selection → Video Restart

## CRITICAL FINDING: Why Videos Stop/Restart When Clicking Channels

The issue stems from a **dependency tracking problem in VideoPlayer's main useEffect hook**.

---

## 1. CLICK HANDLER CHAIN

### User Clicks ChannelCard → ChannelGallery → Index.tsx

**ChannelCard.tsx (Line 50):**
- onClick={onSelect} triggers callback

**ChannelGallery.tsx (Lines 147-150):**
- handleChannelSelect finds index and passes it up

**Index.tsx (Lines 231-235):**
- handleSelectChannel updates setCurrentIndex(index) and setViewMode('player')
- This is the KEY STATE CHANGE

---

## 2. CURRENT CHANNEL RESOLUTION FLOW

**Index.tsx (Lines 251-258):**
- currentChannelRef.useRef created to track channel without re-renders
- useEffect dependency: [currentIndex, channels]
- When currentIndex changes, currentChannelRef.current = channels[currentIndex]
- currentChannel prop read from ref

**VideoPlayer receives:**
- channel={currentChannel} prop passed down
- This reference change triggers the HLS effect

## 3. VIDEOPLAYER HLS LOADING - THE PROBLEM ⚠️

**VideoPlayer.tsx (Lines 64-232):**

DEPENDENCY ARRAY: }, [channel?.url]);
- This triggers WHENEVER channel.url changes

IMMEDIATE STATE RESETS (Lines 70-73):
- setIsLoading(true)
- setShowLoadingOverlay(true)
- setError(null)
- setRetryCount(0)

STREAM CLEANUP (Lines 86-92):
- hlsRef.current.destroy()     ← HLS DESTROYED

- overlayTimeoutRef: 8s timeout to hide loading overlay
- loadTimeoutRef: 30s timeout to show "stream taking too long" error

NEW HLS INSTANCE (Lines 101-131):
- New Hls() object created with configuration
- hls.loadSource(channel.url)  ← NEW STREAM LOADED
- hls.attachMedia(video)

PLAYBACK RESUME (Lines 132-148):
- Waits for Hls.Events.MANIFEST_PARSED
- setIsLoading(false)
- video.play()                 ← VIDEO RESTARTS HERE ⚠️

## 6. ROOT CAUSE ANALYSIS

PRIMARY ISSUE:
- useEffect dependency [channel?.url] triggers full reload on ANY channel change
- video.pause() is called immediately (line 92)
- Complete HLS destruction causes playback interrupt
- User sees 100-8000ms loading delay depending on stream

SECONDARY ISSUES:
- Retry logic adds 2s-4s per failed attempt (2 retries max)
- 30s timeout error if stream doesn't load
- Health check filtering (Index.tsx:150-161) may change channels array
- Every array change might update currentChannelRef reference

## 8. EXACT DEPENDENCY TRACKING

VideoPlayer.tsx Line 232:
  }, [channel?.url]);

This means effect re-runs when:
- channel becomes non-null (from null)
- channel.url changes to different URL
- channel object reference changes but url is same ← POTENTIAL ISSUE

This causes unnecessary resets if channel reference changed but URL identical.

## 11. WHY IT FEELS LIKE "STOP/RESTART"

Immediate perception:
1. Click new channel
2. Loading overlay appears (from setShowLoadingOverlay)
3. Video pauses (from video.pause at line 92)
4. Visual freeze for 100ms-8s depending on stream load
5. Video auto-plays when manifest loads
6. User sees: Stop → Loading → Restart

Actual technical process:
- Old HLS destroyed
- New HLS created
- New manifest downloaded
- New stream buffered
- Video resumed

This is BY DESIGN for reliability, but creates jarring UX perception.
